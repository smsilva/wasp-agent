# SBOM — Software Bill of Materials

**Date:** 2026-05-26  
**Status:** Idea  

## Contexto

Um **SBOM** é um inventário formal e legível por máquina de todos os componentes de software em um produto — dependências diretas, transitivas, bibliotecas de sistema, e metadados de versão/licença/origem. Analogia: lista de ingredientes de um produto alimentício.

Regulamentações e frameworks emergentes passaram a exigir ou recomendar SBOMs:

- **EO 14028 (EUA, 2021):** exige SBOM para software vendido ao governo federal americano.
- **EU AI Act / Cyber Resilience Act (CRA, 2024):** CRA exige SBOMs para produtos digitais vendidos na UE com elementos de conectividade.
- **NTIA Minimum Elements (2021):** define campos mínimos: nome, versão, fornecedor, identificador único, relacionamentos de dependência, autor do SBOM, timestamp.

Formatos padrão:

| Formato | Mantido por | Uso típico |
|---|---|---|
| **SPDX** (ISO/IEC 5962:2021) | Linux Foundation | Conformidade regulatória, OSS |
| **CycloneDX** | OWASP | Segurança, análise de risco |
| **SWID** | ISO/IEC 19770-2 | Governança corporativa |

## Por que interessa ao wasp-agent

O agente provisiona plataformas com dependências Python (`uv`/`pyproject.toml`) e imagens de container. Em dois eixos:

1. **wasp-agent em si:** expõe suas próprias dependências como SBOM — útil para auditoria de licenças, detecção de CVEs, conformidade futura com CRA.
2. **Plataformas provisionadas:** o agente poderia gerar/atualizar SBOMs dos workloads que provisiona — adicionando o SBOM ao repo GitOps como artefato de conformidade.

## Conceitos-chave

### Tipos de conteúdo

- **Dependências de software:** pacotes PyPI, npm, go modules, imagens base.
- **Licenças:** MIT, Apache-2.0, GPL — conflitos de licença são risco legal.
- **CVEs conhecidos:** um SBOM + VEX (Vulnerability Exploitability eXchange) diz quais vulnerabilidades afetam *este* produto e se são exploráveis.
- **Proveniência:** onde o componente foi obtido (PyPI, GitHub, registry privado).

### VEX (Vulnerability Exploitability eXchange)

Complemento ao SBOM: documenta, para cada CVE relevante, se o produto é de fato afetado. Evita falsos positivos em scanners. Formato CycloneDX suporta VEX nativamente.

### Supply chain attacks

SBOMs não previnem ataques, mas permitem **resposta rápida**: quando um CVE como Log4Shell aparece, quem tem SBOM identifica exposição em minutos em vez de dias.

## Ferramentas relevantes

| Ferramenta | Função |
|---|---|
| `syft` (Anchore) | Gera SBOM de imagens, diretórios, lockfiles. Suporta SPDX e CycloneDX. |
| `grype` (Anchore) | Escaneia SBOM por CVEs. Integra com `syft`. |
| `trivy` (Aqua) | Escaner all-in-one: SBOM + CVE + config + secrets. |
| `cdxgen` (OWASP) | Gera CycloneDX de múltiplos ecossistemas. |
| `pip-audit` | CVE scan específico para dependências Python. |
| Dependabot / Renovate | Atualização automática de dependências com alerta de CVE. |

Para Python + `uv`, `syft` lê `pyproject.toml` e `uv.lock` diretamente.

## Direção

### Ação mínima (wasp-agent em si)

1. Adicionar target `make sbom` que roda `syft` no projeto e salva `sbom.cdx.json` (CycloneDX).
2. Adicionar `make scan-vulns` que roda `grype sbom:sbom.cdx.json` e reporta CVEs críticos/altos.
3. Incluir `sbom.cdx.json` como artefato no CI (não commitar no repo — gerar on-demand).

### Expansão para plataformas provisionadas

- No fluxo de provisioning (`wasp/provision.py`), após criar o repo GitOps, gerar SBOM do workload e fazer commit como `sbom/sbom.cdx.json` no repo da plataforma.
- `wasp/watcher.py` pode monitorar atualizações de SBOM (novo commit em `sbom/`) e notificar o usuário se novos CVEs críticos forem detectados.

## Conexão com outros specs

- **EU AI Act (`2026-05-26-eu-ai-act.md`):** CRA (Cyber Resilience Act, irmão do AI Act para produtos digitais) exige SBOM. Mesmo que `wasp-agent` não esteja em alto risco hoje, SBOM é fundamento de conformidade.
- **DORA Metrics (`2026-05-26-dora-metrics.md`):** CFR (Change Failure Rate) pode ser correlacionado com introdução de dependências com CVEs — SBOM fornece o dado.
- **Token/cost budget (`2026-05-20-token-cost-budget.md`):** sem conexão direta, mas ambos são instrumentação de observabilidade do sistema.

## Armadilhas

- **SBOM como checkbox.** Gerar e ignorar não serve. Valor real é no loop: gerar → escanear → agir (update ou VEX).
- **Falsos positivos não tratados.** CVEs em dependências transitivas que não são exercitadas viram ruído. VEX resolve.
- **Formato errado para o contexto.** SPDX para conformidade regulatória; CycloneDX para segurança operacional. Escolher antes de implementar.
- **Lockfile ausente.** `syft` precisa de `uv.lock` (ou equivalente) para dependências transitivas. Sem lockfile, SBOM fica incompleto.
